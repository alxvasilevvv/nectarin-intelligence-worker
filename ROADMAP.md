# NECTARIN Intelligence — глобальный план (Marketing OS)

> Цель: стать **операционной системой маркетинга** для RU/CIS — один AI-коннектор,
> который закрывает работу всей команды (22+ профессии), а установка, доступ, учёт и
> монетизация идут **через единую точку входа — Unyly**. NECTARIN = хаб; Unyly = шлюз,
> маркетплейс и биллинг.

Все данные — demo/синтетика, пока не подключены данные клиента. Расчёты —
детерминированные модели (не «галлюцинации» LLM), к каждому числу есть методология.

---

## 1. Стратегические столбы

1. **Ширина** — покрытие профессий. 22 роли сегодня → дорожная карта до 30+.
   `role_playbook` как адопшн-движок: сотрудник называет роль → получает тулкит, поток
   работы и KPI за один запрос.
2. **Глубина** — за каждым ответом точная модель (97 инструментов), а не общий совет.
3. **Сеть** — ценность растёт с каждым подключённым сотрудником и с каждым
   подключённым к хабу специализированным MCP (федерация, см. §4).
4. **Дистрибуция и монетизация — через Unyly** — единственная точка входа: OAuth 2.1,
   изоляция тенантов, метеринг, тарифы, биллинг по факту.

Аргумент «в 100 раз больше» = Ширина × Глубина × Сеть (оценка эффекта, не гарантия).

## 2. Текущее состояние (готово)

- **97 инструментов / 72 промпта**, 22 профессии, `role_playbook`.
- **Дистрибуция через Unyly**: `connect_via_unyly` (трекнутые установки, UTM, `via=`).
- **Монетизация (реальная, не нарратив)**:
  - plan-gating флагманских инструментов по тарифу (claim `plan` от Unyly Connect);
  - месячные квоты free-tier (100 запросов/мес), CTA на апгрейд через Unyly;
  - метеринг каждого `tools/call` (структурный `usage`-лог + KV-счётчик по тенанту);
  - `GET /usage` — потребление по тенанту;
  - Unyly-атрибуция (footer + `poweredBy`) в выводах.

## 3. Дорожная карта (фазы)

**Статус:** Фазы A–C поставлены; Фаза C-runtime закрыта (v2.53). Активна Фаза D
(v2.54 — расширение профессий: maturity-скоринг, martech-ROI, Van Westendorp PSM).

### Фаза A — «Прозрачность потребления» (v2.50) ✅
- **Заголовки квоты** в ответах `tools/call`: `X-Plan`, `X-Quota-Limit`, `X-Quota-Used`,
  `X-Quota-Remaining`, `X-Quota-Period` — клиент видит остаток без отдельного запроса.
- **HTML-дашборд** `/dashboard` поверх `/usage` — наглядное потребление для руководства.

### Фаза B — «Скилы и расширяемость» (v2.51) ✅
- **Слой скилов** (`src/skills.ts`): именованные композитные воркфлоу, которые
  объединяют несколько инструментов в один сценарий («скил») с готовым результатом.
  Каталог скилов + запуск по имени. Это «дополняемая» прослойка поверх инструментов.
- **Новые модели** (расширение функционала): добор сильных профессиональных инструментов
  (например, retention-когорты для продакта, медиа-микс калибровка, и т.п.).

### Фаза C — «Федерация MCP через Unyly» (v2.52–v2.60) ✅ LIVE
- NECTARIN как **хаб**: помимо собственных инструментов, по запросу подключает
  лучшие специализированные MCP-сервера через Unyly-маркетплейс (поиск ключевых слов,
  биржевые данные, креатив-генерация, веб-аналитика и т.п.).
- Слой **`mcp_federation`** (v2.52): реестр доверенных внешних MCP (через Unyly), единый
  каталог возможностей, маршрутизация запроса в нужный сервер, единый учёт на gateway.
- **C-runtime** (v2.53): **`federation_invoke`** — реальный прокси одного `tools/call`
  во внешний MCP *через* NECTARIN. Fail-closed без конфига — трекнутая ссылка и
  **никаких** сетевых вызовов. Только ключи из реестра (без произвольных URL ⇒ без SSRF).
- **C-gateway** (v2.60): **живой шлюз Unyly** — при `UNYLY_GATEWAY_TOKEN` (самовыпуск
  в dashboard Unyly) NECTARIN проксирует на `gateway.unyly.org/mcp/<slug>` одним токеном
  для всех серверов, без ручной настройки URL на каждый MCP. Per-server `FED_<KEY>_URL`
  по-прежнему переопределяет шлюз. Health: `GET …/mcp/ping`.
- Принцип: **всё потребление и установки — через unyly.org**, поэтому подключение
  внешних серверов тоже монетизируется и измеряется в одной точке.

## 4. Федерация MCP (как «брать всё лучшее из мира ИИ-маркетинга»)

Архитектурно NECTARIN остаётся единым входом для пользователя, а под капотом —
оркестратор, который может звать внешние специализированные MCP-сервера, опубликованные
в Unyly. Дизайн (для Фазы C):

```
Пользователь (Claude/Cursor)
        │
        ▼
NECTARIN Intelligence (хаб)  ──►  собственные 102 инструмента
        │
        └─►  Unyly gateway  ──►  внешние MCP (SEO-данные, биржи, креатив, аналитика…)
                   │
                   └─ единый OAuth, метеринг, биллинг, изоляция
```

Выгода владельцу Unyly: каждый внешний вызов проходит через шлюз → трафик и
монетизация концентрируются на `unyly.org`.

## 5. Принципы реализации

- Любая монетизационная фича **не ломает** текущий authless-прод: gating/квоты неактивны
  без claim `plan` (claimless ⇒ `owner` ⇒ безлимит).
- Каждый релиз: typecheck → тесты → деплой → smoke → commit/push.
- Детерминированные модели, методология и дисклеймеры у каждого числа.
- Документация (README/USAGE/OVERVIEW/CHANGELOG) синхронна с кодом.
